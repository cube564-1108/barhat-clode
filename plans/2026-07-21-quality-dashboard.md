# Quality Dashboard — отчет по качеству сборки букетов

**Статус:** [ ] В процессе
**Дата:** 2026-07-21
**Цель:** Создать аналитический dashboard для мониторинга качества сборки букетов по салонам, флористам и видам заказов с данными из Pyrus

---

## Источники данных

**Pyrus API:**
- Форма: «Оценка качества сборки букетов» (ID: 1327961)
- Поля группировки: Салон, Флорист, Вид заказа
- Авторизация: API токен

**Критерии оценки (9 шт, шкала 0-2):**
1. Соответствие каталогу
2. Аккуратность упаковки
3. Оформление клубники
4. Обработка цветка
5. Свежесть компонентов
6. Техника сборки
7. Клубника отделена от цветка прозрачной пленкой
8. Соответствие правилам вложения материалов
9. Фотография

**Нормировка по видам заказов:**
| Вид заказа | Макс. балл |
|---|---|
| Клубничный букет | 14 |
| Цветочный букет | 14 |
| Коробочка с клубникой или бананами | 14 |
| Клубнично-цветочный букет | 18 |
| Цветочный бокс | 14 |
| Коробочка+цветочный букет | 18 |
| Клубничный бокс | 14 |
| Цветочно-клубничный бокс | 18 |

**Формула качества:** (сумма баллов) / (макс. балл для вида) → 0-1 (или 0-100%)

---

## Архитектура решения

**Технологический стек:**
- Backend: Python (Flask/FastAPI)
- Frontend: HTML + Chart.js (или Plotly Dash)
- База данных: SQLite (локальное кеширование)
- Хостинг: Amvera
- Доступ: Публичный → авторизация позже

**Компоненты:**
1. **Pyrus API Client** — загрузка задач по форме
2. **Data Processor** — нормировка, агрегация, кеширование
3. **Dashboard** — визуализация и фильтры
4. **Scheduler** — обновление данных (cron/trigger)

## Критика фаз (Чат 3)

### 🔴 Критические проблемы

**1. Дублирование данных (Фаза 1 + Фаза 2)**
- Загружаем из Pyrus → SQLite → агрегация → ещё таблицы
- **Упрощение:** SQLite оставить (для оффлайн-работы), но убрать материализованные представления

**2. Heatmap избыточна (Фаза 3.5)**
- Сложный UI, малоинформативна для одного владельца
- **Упрощение:** Заменить на таблицу "Слабые критерии по салонам"

**3. Графики с зумом (Фаза 3.6.4)**
- Избыточно для dashboard'а
- **Упрощение:** Убрать зум, группировка по неделям фиксированная

### 🟡 Конфликты

**4. Фаза 4 vs Фаза 5 — фильтры vs алерты**
- Алерты глобальные, фильтры только для просмотра

**5. Фаза 6 vs Фаза 7 — деплой до тестов**
- Перенести 7.1-7.2 в Фазу 5 (до деплоя)

### 🔐 Безопасность

**6. Публичный доступ без авторизации**
- **Решение:** Добавить простой secret header для защиты

**7. Pyrus API rate limits**
- **Решение:** Лимит 1 запрос/мин для автообновления
